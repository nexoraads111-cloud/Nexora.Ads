1. БАЗОВАЯ КОНЦЕПЦИЯ: КАК РАБОТАЕТ ВЕБ
--------------------------------------------------------------------------------

Что такое сайт на самом деле:
  Браузер (Chrome) → запрашивает файл с сервера → получает HTML/CSS/JS → рисует страницу

Роли технологий (запомни раз и навсегда):
  HTML  = структура (заголовки, кнопки, формы, секции)

У тебя гибрид:
  - Фронт на GitHub Pages (index.html)
  - Бэкенд на Vercel: https://ainexora.vercel.app/api

2. REQUEST / RESPONSE — СЕРДЦЕ ВЕБА
--------------------------------------------------------------------------------

Каждый раз, когда сайт что-то отправляет или получает — это HTTP-запрос.

Request (запрос):
  - method: GET (получить) или POST (отправить данные)
  - url: куда стучимся (например /api/send-application)
  - headers: метаданные (Content-Type: application/json)
  - body: данные (JSON с именем, контактом, сообщением)

Response (ответ):
  - status: 200 OK, 400 ошибка клиента, 500 ошибка сервера
  - body: JSON с результатом или текстом ошибки

Где это у тебя в коде:
  fetch(`${API_BASE}/send-application`, { method:'POST', ... })
  → заявка уходит на Vercel-бэкенд
  → менеджер получает в Telegram

Firebase onSnapshot — это не обычный request, а «подписка»: отзывы обновляются в реальном времени.

Задачи:
[ ] В Network (F12) отправить заявку и увидеть запрос POST к /api/send-application
[ ] Понять: почему при ошибке API открывается Telegram (fallback в sendApplication)
[ ] Выучить: GET vs POST, JSON, CORS (почему фронт и API на разных доменах)
[ ] Прочитать про статус-коды: 200, 401, 403, 404, 500

3. GIT — НЕ КОПИРОВАТЬ ВРУЧНУЮ НА GITHUB
--------------------------------------------------------------------------------

Проблема сейчас: правишь в VS Code → копируешь → вставляешь на github.com
Это опасно: легко затереть чужие правки, потерять историю, сломать прод.

Git = система версий. GitHub = облако для репозитория.

Минимум, который ОБЯЗАН знать:

  git status          — что изменилось
  git diff            — какие именно строки
  git add .           — подготовить файлы к коммиту
  git commit -m "..." — сохранить снимок с описанием
  git push            — отправить на GitHub
  git pull            — забрать изменения с GitHub

Ветки (branches):
  main/master — продакшен (то, что видят клиенты)
  feature/... — эксперименты, не ломаешь основной сайт

Задачи:
[ ] Установить Git, настроить VS Code: Source Control (иконка ветки слева)
[ ] Сделать 5 коммитов через VS Code, НЕ через копипаст на сайте
[ ] Научиться: git pull перед работой, git push после
[ ] Понять .gitignore — что НЕ должно попадать в репозиторий
[ ] Прочитать про Pull Request — как принято вносить изменения в команде

4. .ENV — СЕКРЕТЫ И НАСТРОЙКИ (КРИТИЧНО!)
--------------------------------------------------------------------------------

Проблема в твоём коде СЕЙЧАС:
  - Firebase config захардкожен прямо в index.html (apiKey, projectId...)
  - API_BASE = 'https://ainexora.vercel.app/api' — тоже в коде
  - Любой может открыть исходник и увидеть ключи

.env — файл с переменными окружения. НЕ коммитится в Git.

Пример .env (локально, на сервере — в настройках Vercel):
  VITE_API_BASE=https://ainexora.vercel.app/api
  VITE_FIREBASE_API_KEY=...
  VITE_FIREBASE_PROJECT_ID=...

Правила:
  1. Создать .env — положить секреты
  2. Добавить .env в .gitignore
  3. Создать .env.example — шаблон БЕЗ реальных ключей (для других разработчиков)
  4. На Vercel: Settings → Environment Variables — те же имена

Важно:
  - На чистом HTML без сборщика .env не работает «из коробки»
  - Нужен либо Vite/Webpack, либо бэкенд отдаёт публичный конфиг
  - Firebase apiKey для клиента — не совсем секрет, но projectId + rules важнее
  - Telegram bot token, пароли, приватные ключи — НИКОГДА в index.html

Задачи:
[ ] Создать .gitignore с .env
[ ] Вынести API_BASE и Firebase config из index.html (после настройки сборки)
[ ] На Vercel добавить переменные окружения для бэкенда
[ ] Понять разницу: публичные vs приватные ключи

  ├── .gitignore
  └── README.md

6. ФРЕЙМВОРКИ: VUE И REACT (следующий уровень)
--------------------------------------------------------------------------------

Зачем фреймворк, если уже есть HTML:
  - Компоненты: Header, Hero, Pricing — переиспользуешь, не копируешь
  - Состояние: язык, модалки, отзывы — без ручного querySelector везде
  - Сборка: .env, минификация, быстрая загрузка
  - Масштаб: когда сайтов станет 10, а не 1

Vue 3 — проще войти, похож на обычный HTML:
  - Один файл .vue = template + script + style
  - Хорош для лендингов и небольших студийных сайтов

React — популярнее в индустрии, больше вакансий:
  - JSX вместо HTML
  - Экосистема огромная (Next.js для SEO-сайтов)

Для NexoraAds логичный путь:
  1. Сначала разделить чистый HTML/CSS/JS (блок 5)
  2. Потом Vite + Vue 3 ИЛИ Vite + React — на выбор
  3. Деплой: Vercel / Netlify (не только GitHub Pages)
